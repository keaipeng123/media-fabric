# 多路 I/O 复用笔记

> 这份笔记整理自前面对以下问题的讨论：
>
> - 什么是多路 I/O 复用
> - select、poll、epoll 的区别
> - fd 是什么
> - 为什么 select、poll、epoll 属于同步 I/O，而不是异步 I/O

---

## 1. 什么是多路 I/O 复用

多路 I/O 复用，本质上是：

- 一个线程同时监听多个文件描述符 fd
- 哪个 fd 就绪了，再去处理哪个 fd

这里的“就绪”通常指：

- 可读
- 可写
- 异常
- 有新的连接到来

它解决的核心问题不是“让单次 I/O 更快”，而是：

- 让一个线程高效管理很多连接
- 避免一个连接对应一个线程带来的线程切换和资源开销

如果没有多路复用，常见做法往往是：

- 一个连接一个线程
- 每个线程阻塞在自己的 socket 上

当连接数很多时，这种模型的成本很高。多路 I/O 复用就是把“等谁先准备好”这件事交给内核统一处理。

---

## 2. 多路 I/O 复用的基本流程

可以把整体流程理解成下面几步：

1. 应用程序把多个 fd 交给内核监听
2. 内核等待这些 fd 的状态变化
3. 某些 fd 准备好以后，内核通知应用程序
4. 应用程序再对这些就绪 fd 做真正的读写处理

所以它的核心思路是：

- 先等事件
- 再处理事件

---

## 3. fd 到底是什么

fd 是 file descriptor，也就是文件描述符。

你可以把它理解成：

- 进程打开一个内核 I/O 对象后
- 内核返回给进程的一个整数编号
- 这个编号就是 fd

后续无论是读、写、关闭，还是加入 select、poll、epoll 监听，都是通过这个编号完成的。

### 3.1 fd 不只是“文件”

虽然名字里叫文件描述符，但它不只表示普通文件，也可以表示：

- 普通文件
- socket
- 管道
- 终端
- 设备

所以更准确地说，fd 表示的是：

- 进程访问某个内核 I/O 资源时使用的句柄编号

### 3.2 常见默认 fd

Linux 里常见的三个默认文件描述符是：

- 0：标准输入
- 1：标准输出
- 2：标准错误输出

### 3.3 放到网络编程里怎么理解

在网络编程中：

- 调用 socket 会得到一个 fd
- 调用 accept 后，新连接也会得到一个新的 fd
- epoll、poll、select 监听的就是这些连接 fd

所以多路 I/O 复用，本质上就是：

- 同时管理很多个 fd
- 谁就绪就处理谁

---

## 4. select 是什么

select 是最经典、最早期的多路复用接口。

它的基本思路是：

1. 准备若干个 fd 集合
2. 把你关心的 fd 放进去
3. 调用 select 等待事件发生
4. select 返回后，检查哪些 fd 就绪了
5. 对这些就绪 fd 进行处理

### 4.1 select 的特点

- 使用 fd 集合表示监听对象
- 每次调用都要把整批 fd 传给内核
- 调用结束后，集合会被修改，下次还要重新设置
- 内核和用户态通常都要遍历一遍 fd 集合

### 4.2 select 的缺点

- fd 数量通常有限制
- 每次都需要重建监听集合
- 需要线性扫描，复杂度通常是 O(n)
- 当连接很多时，效率会下降明显

适用场景：

- 连接数量不多
- 兼容性要求高
- 教学或简单程序

---

## 5. poll 是什么

poll 可以看作 select 的改良版。

它不再使用位图式集合，而是使用结构体数组描述每个 fd 及其关注的事件。

### 5.1 poll 的优点

- 没有 select 那种明显的固定小上限
- 事件表达方式更直接
- 使用上比 select 更自然一些

### 5.2 poll 的问题

虽然 poll 解决了 select 的部分限制，但它的核心问题没有变：

- 每次调用仍然要把全部 fd 传给内核
- 内核仍然要遍历整个 fd 数组
- 用户程序通常也要再遍历返回结果

所以 poll 的性能瓶颈依然是：

- 当监听 fd 数量很大时，仍然需要 O(n) 级别扫描

适用场景：

- 连接数中等
- 不想用 select 的位图接口
- 但对极限高并发没有很高要求

---

## 6. epoll 是什么

epoll 是 Linux 下高并发服务器最常用的多路复用机制。

它和 select、poll 最大的区别是：

- 不是每次都把所有 fd 拿去问一遍
- 而是把关注的 fd 长期注册到内核里
- 真正有事件发生时，内核把就绪 fd 放到就绪队列
- 应用调用 epoll_wait 时，直接拿到就绪结果

### 6.1 epoll 的典型接口

- epoll_create：创建 epoll 实例
- epoll_ctl：注册、修改、删除监听 fd
- epoll_wait：等待并获取就绪事件

### 6.2 epoll 为什么更适合高并发

原因主要有：

- fd 只需要注册一次，不用每次全量传入
- 返回的是已经就绪的 fd，而不是所有被监听的 fd
- 更适合“连接很多，但同一时刻活跃连接不多”的服务器场景

可以把它理解成：

- select、poll 更像“整表轮询”
- epoll 更像“事件驱动通知”

---

## 7. select、poll、epoll 的核心对比

### 7.1 fd 数量支持

- select：通常有上限，限制更明显
- poll：比 select 更灵活
- epoll：适合大规模 fd 监听

### 7.2 时间复杂度

- select：通常是 O(n)
- poll：通常也是 O(n)
- epoll：更接近和活跃 fd 数量相关，不必每次遍历全部监听 fd

### 7.3 用户态和内核态之间的数据传递

- select / poll：每次调用通常都要重新传入整批 fd
- epoll：fd 在注册后由内核维护，等待阶段不需要每次全量重传

### 7.4 适用场景

- select：简单、小规模、兼容性好
- poll：中等规模，接口比 select 灵活
- epoll：Linux 高并发服务器首选

---

## 8. 为什么说 select、poll、epoll 都是同步 I/O

这是最容易混淆的点。

先说结论：

- select、poll、epoll 负责的是“通知你哪个 fd 已经就绪”
- 但真正把数据读出来或写进去，仍然要应用程序自己调用 read、recv、write 等函数

也就是说：

- 它们只负责“就绪通知”
- 不负责替你完成真正的读写操作

所以它们属于同步 I/O，而不是异步 I/O。

### 8.1 一次读操作可以粗略拆成两步

以 socket 读数据为例，可以粗略拆成两个阶段：

1. 等待数据到达内核
2. 把数据从内核空间拷贝到用户空间缓冲区

多路复用机制做的事情主要是：

- 帮你等待“这个 fd 什么时候可以读”

但真正的数据搬运，还是要你自己调用 recv 或 read 去完成。

这就是它仍然属于同步 I/O 的根本原因。

---

## 9. 同步 I/O 和异步 I/O 到底是什么意思

### 9.1 同步 I/O

同步 I/O 的核心是：

- 真正的 I/O 操作需要应用程序自己主动发起
- 应用程序要亲自调用 read、recv、write 等接口
- 调用返回后，才知道这次 I/O 是否完成

换句话说：

- 内核不会替你把整个 I/O 过程彻底做完再通知你
- 应用只是先知道“现在可以做了”
- 后面的实际读写还要自己完成

### 9.2 异步 I/O

异步 I/O 的核心是：

- 应用先把 I/O 请求交给内核
- 内核不仅等待数据准备好
- 还负责把数据直接拷贝到用户空间缓冲区
- 整个 I/O 全部完成后，再通知应用程序

最关键的一句话是：

- 同步 I/O：通知你可以开始读了
- 异步 I/O：通知你已经读完了

### 9.3 为什么 epoll 仍然不是异步 I/O

以 epoll 为例，实际流程通常是：

1. 调用 epoll_wait
2. 内核告诉你某个 fd 可读了
3. 你自己调用 recv
4. recv 把数据从内核拷贝到你的 buffer
5. recv 返回后，你才真正拿到数据

关键点在第 3 步：

- 真正的读取动作还是你自己调用的
- 内核没有替你把数据直接放进用户缓冲区后再通知你

所以 epoll 只是“就绪通知”，不是“完成通知”。

因此它仍然属于同步 I/O。

---

## 10. 同步/异步 和 阻塞/非阻塞 不是一回事

这是另一个高频混淆点。

### 10.1 阻塞/非阻塞关注什么

阻塞和非阻塞描述的是：

- 一个调用发起后，当前线程会不会被挂起等待

例如：

- 阻塞 recv：没数据时线程会卡住
- 非阻塞 recv：没数据时立刻返回

### 10.2 同步/异步关注什么

同步和异步描述的是：

- 这次 I/O 到底是谁主导完成的
- 应用是不是还需要自己亲手把读写做完

### 10.3 常见组合

- 阻塞同步 I/O：直接调用阻塞式 recv，没有数据就等
- 非阻塞同步 I/O：调用非阻塞 recv，没数据立刻返回
- 多路复用同步 I/O：先通过 select、poll、epoll 等待就绪，再自己调用 recv
- 异步 I/O：应用提交请求后，不需要自己再调 read 取数据，内核完成后再通知

所以不能简单地把概念理解为：

- 阻塞 = 同步
- 非阻塞 = 异步

这是错误的。

---

## 11. epoll 里的 LT 和 ET

这是 epoll 里必须掌握的知识点。

### 11.1 LT：水平触发

Level Trigger 的特点是：

- 只要 fd 里还有数据没处理完
- epoll_wait 之后还会继续通知你

例如：

- socket 缓冲区里有 1000 字节
- 你这次只读了 100 字节
- 还剩 900 字节
- 下次 epoll_wait 仍然会继续提示这个 fd 可读

优点：

- 编程更简单
- 不容易漏事件
- 更适合作为默认模式

### 11.2 ET：边缘触发

Edge Trigger 的特点是：

- 只有状态从“无事件”变成“有事件”时通知一次

例如：

- 原来没有数据
- 现在突然来了 1000 字节
- epoll 通知你一次
- 如果你没有尽量读空，下次可能不再提醒

因此 ET 模式通常要求：

- 使用非阻塞 fd
- 在收到可读事件后循环读取
- 一直读到返回 EAGAIN 或 EWOULDBLOCK 为止

### 11.3 为什么 ET 常配合非阻塞

如果 ET 配合阻塞 socket，可能出现：

- 你在一次事件处理中进入读循环
- 数据已经读完后再次 read
- 结果线程被阻塞住
- 整个事件循环卡死

所以 ET 模式下通常要配合非阻塞 I/O。

---

## 12. 为什么 epoll 更适合高并发

高并发服务器通常有一个典型特征：

- 连接很多
- 但同一时刻真正活跃的连接并不多

例如：

- 有 50000 个连接
- 但当前只有几百个连接在收发数据

这时：

- select / poll 通常仍要遍历大量空闲连接
- epoll 更关注真正就绪的那一小部分 fd

所以在“海量连接、少量活跃”的场景下，epoll 的优势会非常明显。

---

## 13. 常见误区

### 13.1 epoll 一定比 select、poll 快很多

不绝对。

如果监听 fd 本来就很少，比如几十个，差距未必明显。

epoll 的优势主要出现在：

- 连接数很多
- 空闲连接多
- 活跃连接占比较低

### 13.2 epoll 是异步 I/O

不是。

epoll 只是事件通知机制，真正的数据读写仍然要应用程序自己调用 recv、read、write 完成，所以它属于同步 I/O。

### 13.3 用了 epoll 就一定高性能

也不对。

系统性能还取决于：

- 事件循环是否写对
- 是否正确使用非阻塞 I/O
- 是否避免在事件回调里做重业务逻辑
- 是否正确处理锁竞争、粘包拆包、线程池等问题

---

## 14. 一个便于记忆的总结

可以这样记三者的区别：

- select：每次拿着名单去问，谁有事
- poll：名单格式更好了，但还是每次整表问一遍
- epoll：先在内核登记好名单，谁有事内核直接通知

再把同步/异步也压缩成一句话：

- 同步 I/O：通知你可以开始读了
- 异步 I/O：通知你已经读完了

---

## 15. 最后一句话总结

多路 I/O 复用就是：

- 用一个线程同时监听多个 fd
- 哪个 fd 就绪了就处理哪个 fd

其中：

- select：老、通用，但有限制
- poll：更灵活，但仍然需要线性扫描
- epoll：Linux 下高并发服务器最常用

而它们都属于同步 I/O，因为它们只负责告诉你“现在可以读写了”，真正的 read、recv、write 仍然需要程序自己去调用。
