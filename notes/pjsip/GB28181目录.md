# GB28181 目录（Catalog）笔记

1. **协议层面**：上下级之间“目录查询/目录响应”的常见交互与字段。
2. **工程实现**：本项目在上级侧如何发起 `Catalog` 查询（请求代码落点与关键点）。

---

## 1. 交互概览（上下级）

![请求gb目录上下级交互流程](../assets/catalog_1.png)

常见流程可以理解为：

- 上级平台向下级平台/设备发送 SIP `MESSAGE`，body 为 GB28181 XML：`<Query><CmdType>Catalog</CmdType>...</Query>`。
- 下级对该 `MESSAGE` 先回 SIP 层的响应（通常 `200 OK`）。
- 下级随后用新的 SIP `MESSAGE`（或按对端约定的方式）上报目录响应（通常是 `<Response>...<CmdType>Catalog</CmdType>...</Response>`），其中包含设备/通道列表。

说明：不同设备/平台在“目录响应是否分包、是否分页、字段是否齐全”等细节上会有差异，以抓包与对端实现为准。

## 2. DeviceID（20 位）组成速记

![gb 设备 id 组成](../assets/catalog_2.png)

- 行政区域编码（常见写法为前 6 位；也有实现按“前 8 位”描述/使用）
- 行业编码（常见为 2 位，例如 `51` 表示金融行业编码的说法）
- 类型编码（常见为 3 位，例如摄像机 `131`，中心信令服务器 `200`）
- 序号编码（剩余若干位，用于唯一编号）

## 4. 工程实现：发起目录查询（Catalog Query）

### 4.1 代码位置

- 头文件：[SipSupService/include/GetCatalog.h](../../SipSupService/include/GetCatalog.h)
- 实现：[SipSupService/src/GetCatalog.cpp](../../SipSupService/src/GetCatalog.cpp)

`GetCatalog::DirectoryGetPro(...)` 的职责：

- 遍历下级列表（`GlobalCtl::getSubDomainInfoList()`）
- 对已注册的下级（`iter->registered`）逐个发送 `Catalog` 查询

### 4.2 SIP 请求构造（关键点）

工程里是手工拼 `MESSAGE`：

- `From/To/Request-URI`：通过 `SipMessage` 组装，使用本机 `sipId/sipIp` 与下级 `sipId/addrIp/sipPort`
- `Method`：`MESSAGE`（通过 `PJSIP_OTHER_METHOD`）
- body：`Content-Type: Application/MANSCDP+xml`，内容是 GB28181 XML

### 4.3 XML Body（Query/Catalog）形态

代码生成的核心结构是：

```xml
<?xml version="1.0"?>
<Query>
	<CmdType>Catalog</CmdType>
	<SN>123</SN>
	<DeviceID>34020000002000000001</DeviceID>
</Query>
```

字段含义：

- `CmdType=Catalog`：表示“目录查询”
- `SN`：序列号/流水号（工程里用 `random()%1024`）
- `DeviceID`：这里填的是“下级的 sipId”（`iter->sipId`）

### 4.4 关键代码片段（便于回忆）

建议把下面几行当作“构造 GB28181 `MESSAGE` 的最小闭环”：

```cpp
XmlParser parse;
auto* rootNode = parse.AddRootNode((char*)"Query");
parse.InsertSubNode(rootNode, (char*)"CmdType", (char*)"Catalog");
parse.InsertSubNode(rootNode, (char*)"SN", tmpStr);
parse.InsertSubNode(rootNode, (char*)"DeviceID", iter->sipId.c_str());

char* xmlbuf = new char[1024];
memset(xmlbuf, 0, 1024);
parse.getXmlData(xmlbuf);

pj_str_t type = pj_str("Application");
pj_str_t subtype = pj_str("MANSCDP+xml");
pj_str_t xmldata = pj_str(xmlbuf);
tdata->msg->body = pjsip_msg_body_create(tdata->pool, &type, &subtype, &xmldata);
```

发送：

- 调用 `pjsip_endpt_send_request(...)` 发送请求
- 代码注释里强调：此处更关心“下级推送的实际节点目录”，不强依赖 `200 OK` 回调

## 5. 现实现的注意点（排障/工程化时很有用）

- **内存泄漏**：`xmlbuf` 使用了 `new char[1024]`，当前函数末尾没有 `delete[]`。
- **锁粒度偏大**：`DirectoryGetPro` 在持有 `GlobalCtl::globalLock` 的情况下进行网络发送，容易放大锁竞争（可考虑“先拷贝列表/必要字段 → 解锁 → 再逐个发送”。）
- **SN 取值范围较小**：`random()%1024` 容易重复；若对端用 `SN` 做严格匹配，建议改成递增/时间戳/原子计数。
- **构造即发送**：`GetCatalog::GetCatalog()` 里直接调用 `DirectoryGetPro(NULL)`，意味着对象一创建就开始发目录请求；排查“为何启动就发 Catalog”时要记得这一点。
