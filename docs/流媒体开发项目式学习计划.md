# 流媒体开发项目式学习计划

更新日期：2026-06-29

## 目标

基于现有经历，从“软交换 / 融合通信后端开发”逐步补齐到“流媒体开发工程师”的能力栈。

当前优势：

- Go / C++ 后端开发经验
- LiveKit / WebRTC 项目经验
- FreeSWITCH / SIP / VoLTE / GB/T 28181 项目经验
- RTP/RTCP、PS、H.264/H.265、epoll/socket 的初步实现经验
- 具备真实业务系统开发、联调、排障和交付经历

当前需要补强：

- FFmpeg / libavformat / libavcodec 实战
- 音视频封装、转封装、转码、录制、切片
- WebRTC 底层：SDP、ICE、SRTP、RTCP Feedback、拥塞控制
- 流媒体服务器体系：RTMP、HTTP-FLV、HLS、WebRTC、SRT
- 工程化：CMake、Docker、压测、抓包、内存与线程问题排查
- 将项目沉淀成作品集、技术文档和面试可讲的架构故事

## 总体路线

推荐路线：

```text
GB/T 28181
  -> RTP/RTCP
  -> PS 解封装
  -> H.264/H.265 裸流
  -> FFmpeg / GStreamer
  -> RTMP / HTTP-FLV / HLS / WebRTC
  -> 录制、回放、低延迟播放、稳定性治理
```

学习方法：

- 每个阶段只围绕一个明确作品推进。
- 每个作品都要有 README、架构图、抓包截图、测试命令和问题记录。
- 每周至少输出一篇学习笔记或排障记录。
- 不追求一开始做大而全，先把一条链路跑通，再逐步增强。

## 作品一：GB/T 28181 转流网关

周期：4-6 周

目标：基于现有 media-fabric，做一个可以把国标设备流转成常见播放协议的网关。

作品名称建议：

- `gb-stream-gateway`
- `GB28181 Media Gateway`
- `国标视频流转发网关`

核心链路：

```text
GB/T 28181 设备
  -> SIP INVITE
  -> RTP/PS
  -> PS Demux
  -> H.264/H.265
  -> FFmpeg / GStreamer
  -> RTMP / HTTP-FLV / HLS
```

第一阶段：跑通裸流输出

任务：

- 从 RTP 包中恢复完整 PS 数据。
- 使用现有 PS 解封装逻辑提取 H.264/H.265。
- 将裸流写入 `.h264` 或 `.h265` 文件。
- 使用 `ffplay` 播放验证：

```bash
ffplay -f h264 output.h264
ffplay -f hevc output.h265
```

要学习：

- H.264 Annex-B 格式
- Start Code：`00 00 01` / `00 00 00 01`
- SPS / PPS / IDR
- H.265 VPS / SPS / PPS
- PTS / DTS 基础

验收标准：

- 能从 GB/T 28181 RTP/PS 流中稳定提取可播放裸流。
- README 记录一次完整抓包和裸流播放过程。

第二阶段：转封装为 FLV / MP4

任务：

- 使用 FFmpeg CLI 完成转封装实验。
- 使用 libavformat 写一个最小转封装程序：
  - 输入 H.264/H.265 裸流
  - 输出 FLV 或 MP4
- 处理时间戳、关键帧、extradata。

要学习：

- `AVFormatContext`
- `AVStream`
- `AVPacket`
- `avformat_write_header`
- `av_interleaved_write_frame`
- H.264/H.265 extradata
- MP4 和 FLV 的差异

验收标准：

- 能生成可播放的 `.flv` 和 `.mp4`。
- 能解释为什么 MP4 对时间戳和 extradata 更敏感。

第三阶段：推送到 SRS

任务：

- 用 Docker 启动 SRS。
- 将流推送为 RTMP。
- 使用 HTTP-FLV / HLS 播放。
- 编写启动脚本。

参考链路：

```text
GB28181 -> RTP/PS -> H.264 -> RTMP -> SRS -> HTTP-FLV/HLS
```

要学习：

- RTMP 基础
- FLV Tag
- HLS 切片
- GOP 与延迟
- 直播首屏时间

验收标准：

- 浏览器或播放器可以播放国标设备转出的 HTTP-FLV / HLS。
- README 中包含：
  - 架构图
  - 启动命令
  - 播放地址
  - 延迟测试结果

最终作品展示：

- 一张架构图
- 一段演示视频
- 一份抓包说明
- 一份“从 RTP 到可播放直播流”的技术总结

简历可补充点：

- “实现 GB/T 28181 转流网关，将 RTP/PS 码流解封装为 H.264/H.265，并通过 FFmpeg/SRS 输出 RTMP、HTTP-FLV、HLS，完成端到端播放链路。”

## 作品二：低延迟 WebRTC 播放网关

周期：5-8 周

目标：把 GB/T 28181 或 RTSP 流转成 WebRTC，实现浏览器低延迟播放。

推荐路线：

```text
GB28181 / RTSP
  -> H.264/H.265
  -> RTP Packetize
  -> WebRTC PeerConnection
  -> Browser
```

可选技术方案：

- 方案 A：使用 SRS WebRTC 能力，先跑通整体链路。
- 方案 B：基于 Pion WebRTC 写一个 Go 版网关。
- 方案 C：基于 GStreamer `webrtcbin` 做 pipeline。

建议优先级：

1. SRS 跑通
2. Pion WebRTC 深入
3. GStreamer 扩展

第一阶段：理解 WebRTC 信令

任务：

- 写一个最小 WebRTC 播放 Demo。
- 理解 Offer / Answer。
- 理解 ICE Candidate。
- 使用浏览器 `chrome://webrtc-internals` 排查连接。

要学习：

- SDP
- ICE
- STUN / TURN
- DTLS
- SRTP
- RTP Payload Type

验收标准：

- 可以解释浏览器和服务端如何建立 WebRTC 连接。
- 可以定位 ICE 失败、SDP 不匹配、编码不支持等问题。

第二阶段：接入视频流

任务：

- 将 H.264 Annex-B 转为 WebRTC 可发送格式。
- 处理 SPS / PPS。
- 支持 PLI 后重发关键帧。
- 支持 RTCP Receiver Report。

要学习：

- H.264 RTP Packetization
- FU-A 分片
- STAP-A
- RTCP PLI / FIR / NACK
- jitter buffer
- 关键帧请求

验收标准：

- 浏览器低延迟播放视频。
- 断流后可以重连。
- 浏览器发 PLI 后服务端能触发关键帧逻辑。

第三阶段：稳定性与体验

任务：

- 统计端到端延迟。
- 增加码流状态日志。
- 增加断流重连。
- 增加多路播放测试。

要学习：

- TWCC
- 带宽估计
- 丢包与重传
- jitter、RTT、packet loss
- simulcast / SVC 基础

验收标准：

- 局域网延迟控制在 500ms 左右。
- 支持多路并发播放。
- 有简单监控指标：在线连接数、码率、帧率、丢包率、RTT。

最终作品展示：

- 浏览器播放页面
- WebRTC 内部指标截图
- 低延迟对比测试：HLS vs HTTP-FLV vs WebRTC

简历可补充点：

- “实现 GB/T 28181/RTSP 到 WebRTC 的低延迟播放链路，处理 H.264 RTP 分片、SPS/PPS、RTCP PLI/NACK、断流重连和播放指标统计。”

## 作品三：直播录制与回放系统

周期：4-6 周

目标：实现从实时流到录像文件，再到点播回放的完整链路。

核心链路：

```text
实时流
  -> 转封装
  -> MP4 / HLS
  -> 索引入库
  -> 查询
  -> 回放 / seek
```

第一阶段：MP4 录制

任务：

- 使用 FFmpeg CLI 录制 RTSP/RTMP 流。
- 用 libavformat 实现录制。
- 处理异常断流后的文件收尾。

要学习：

- MP4 moov box
- fMP4
- PTS / DTS
- B 帧
- 音视频同步
- 时间基 `time_base`

验收标准：

- 录制文件可播放。
- 异常中断后尽可能保证文件可恢复。

第二阶段：HLS 切片

任务：

- 生成 `.m3u8` 和 `.ts` 或 fMP4 切片。
- 支持按时间清理旧切片。
- 支持回放指定时间段。

要学习：

- HLS playlist
- TS / fMP4
- segment duration
- keyframe alignment

验收标准：

- 浏览器或播放器可播放 HLS。
- 支持按时间段回放。

第三阶段：录像索引

任务：

- 设计录像表：
  - stream_id
  - start_time
  - end_time
  - file_path
  - duration
  - codec
  - size
- 提供查询接口。
- 提供删除、下载、预览接口。

验收标准：

- 可查询历史录像。
- 可按时间段回放。
- README 有完整 API 示例。

简历可补充点：

- “实现实时流录制与回放链路，支持 MP4/HLS 录制、切片、索引查询、按时间段回放，并处理 PTS/DTS 与异常断流文件恢复问题。”

## 作品四：流媒体诊断工具

周期：2-4 周

目标：做一个命令行工具，用来分析 RTP、PS、H.264/H.265、FLV、MP4、HLS。

作品名称建议：

- `stream-inspector`
- `media-probe`
- `rtp-analyzer`

功能：

- 分析 RTP：
  - SSRC
  - sequence
  - timestamp
  - payload type
  - 丢包
  - 乱序
- 分析 H.264/H.265：
  - NALU 类型
  - SPS/PPS/VPS
  - 分辨率
  - 帧类型
- 分析 FLV：
  - audio/video tag
  - timestamp
  - keyframe
- 分析 HLS：
  - m3u8
  - segment duration
  - 是否关键帧对齐

要学习：

- Wireshark 抓包分析
- 二进制协议解析
- FFmpeg `ffprobe`
- 自定义日志与统计

验收标准：

- 能用命令行分析一个 `.pcap` 或媒体文件。
- 输出可读报告。

示例：

```bash
stream-inspector rtp input.pcap
stream-inspector h264 output.h264
stream-inspector flv live.flv
stream-inspector hls index.m3u8
```

简历可补充点：

- “开发流媒体诊断工具，支持 RTP 丢包/乱序统计、H.264/H.265 NALU 解析、FLV/HLS 基础检测，用于快速定位播放卡顿、黑屏和时间戳异常问题。”

## 面试讲解模板

每个作品都按这个结构准备：

### 1. 项目背景

这个项目解决什么问题？为什么要做？

示例：

> 为了补齐 GB/T 28181 到通用流媒体播放协议的链路，我基于现有 C++ 信令服务实现了一个国标转流网关，将 RTP/PS 流解封装为 H.264/H.265，并输出 RTMP、HTTP-FLV、HLS 和 WebRTC 播放。

### 2. 技术架构

说明输入、处理、输出。

示例：

```text
GB28181 Device
  -> SIP INVITE
  -> RTP/PS
  -> PS Demux
  -> H.264/H.265
  -> FFmpeg
  -> SRS
  -> HTTP-FLV/HLS/WebRTC
```

### 3. 你的职责

必须突出“我做了什么”：

- 信令流程
- RTP 收包
- PS 解封装
- 时间戳处理
- 转封装
- 播放协议输出
- 稳定性治理

### 4. 技术难点

可选难点：

- RTP 丢包与乱序
- PS 解封装残包处理
- H.264 SPS/PPS 提取
- MP4 时间戳
- WebRTC PLI 关键帧请求
- TCP 主被动连接与超时
- 多路并发开流

### 5. 结果

尽量量化：

- 支持几种输出协议
- 延迟多少
- 并发多少路
- 断流重连时间
- CPU / 内存占用

## 注意事项

- 不要只看教程，一定要让数据流真实跑起来。
- 不要绕过抓包，SIP/RTP/WebRTC 问题最终都要靠抓包和日志定位。
- 不要把 FFmpeg 当黑盒，只用命令行不够，至少要写 2-3 个 libav 小程序。
- 不要一开始追求高并发，先保证链路正确，再做稳定性和性能。
- 每完成一个阶段，都要更新 README 和简历草稿。
