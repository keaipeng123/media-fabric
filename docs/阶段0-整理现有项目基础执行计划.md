# 阶段 0：整理现有项目基础执行计划

更新日期：2026-06-29

## 目标

将当前 `GB28181-Server` 整理成后续流媒体项目式学习的工程底座，并把 `SipSupService` 与 `SipSubService` 逐步整合为一个统一项目、一个进程、一个服务入口：

```text
gb28181-server
  -> GB28181Node
      -> SipStack
      -> PeerRegistry
      -> RegisterClientCapability
      -> RegisterServerCapability
      -> KeepaliveCapability
      -> CatalogCapability
      -> InviteCapability
      -> MediaReceiveCapability
      -> MediaSendCapability
```

最终工程标准不是运行时选择启用哪些能力，而是 `gb28181-server` 进程天然具备 GB28181 节点应有的完整能力。上级/下级只表示某条级联关系中的相对身份：对上级平台，本节点表现为注册客户端；对下级平台或设备，本节点表现为注册服务端。代码组织应按能力拆分，而不是按上下级身份拆分。

阶段 0 不追求新增 FFmpeg、SRS、WebRTC 等能力，重点是让现有项目结构、构建、文档和验证链路变清楚。

## 当前整理结果

## 执行原则

- 先统一工程入口，再重构业务代码。
- 先保持行为不变，再抽取公共模块。
- `SipSupService` 和 `SipSubService` 只作为历史代码来源和迁移参照，不作为最终交付形态。
- 阶段 0 的工程主线始终指向单一进程：一个配置、一个日志体系、一个 `GB28181Node`、一组能力模块、一个服务入口。
- 能力模块需要能被同一个节点同时装配，例如注册客户端和注册服务端可以同时存在，媒体接收和媒体发送也可以同时存在。
- 所有阶段性成果都要能被 README、流程图、抓包或命令验证。

## 里程碑 4：节点与能力模块封装

目标：把历史上分散在 `SipSupService` 与 `SipSubService` 的业务逻辑，迁移为同一 `GB28181Node` 下的能力模块。上级/下级作为运行时对端关系处理，工程结构不再按身份拆分主模块。

建议结构：

```text
src/
  core/
    GB28181Node
    NodeConfig
    SipStack
    PeerRegistry
    SessionManager
  capabilities/
    RegisterClientCapability
    RegisterServerCapability
    KeepaliveClientCapability
    KeepaliveServerCapability
    CatalogClientCapability
    CatalogServerCapability
    RecordQueryClientCapability
    RecordQueryServerCapability
    InviteClientCapability
    InviteServerCapability
  media/
    MediaManager
    RtpReceiver
    RtpSender
    PsMuxer
    PsDemuxer
```

任务：

- 将 `SipSupService/src/main.cpp` 中接收注册、发起查询、发起 INVITE、接收媒体等逻辑拆入对应能力模块。
- 将 `SipSubService/src/main.cpp` 中主动注册、心跳、响应查询、响应 INVITE、发送媒体等逻辑拆入对应能力模块。
- `GB28181Node` 负责装配能力、管理生命周期、持有节点配置、对端注册表、会话管理器和媒体管理器。
- 能力模块之间通过明确接口协作，避免继续依赖历史全局控制对象。
- 一个 SIP 监听栈可以承载本节点的所有 SIP 能力；确需多端口时也应作为节点配置，而不是上级/下级模块拆分。

验收标准：

- `gb28181-server` 启动时创建一个 `GB28181Node`，并装配完整能力模块。
- 本节点既能向上级平台注册，也能接收下级平台或设备注册。
- 节点初始化、能力启动、能力停止和资源释放边界清楚。

## 里程碑 5：单一二进制入口

目标：新增统一入口 `gb28181-server`，编译时生成一个服务，运行时固定启动一个具备完整 GB28181 能力的节点。

建议启动方式：

```bash
./gb28181-server -c conf/gb28181-server.conf
```

配置结构建议：

```ini
[node]
sip_id = 11000000002000000001
sip_ip = 127.0.0.1
sip_port = 7101

[peer.upstream.1]
sip_id = 10000000002000000001
sip_ip = 127.0.0.1
sip_port = 5061

[peer.downstream.1]
sip_id = 12000000002000000001
sip_ip = 127.0.0.1
sip_port = 7201

[media]
rtp_port_begin = 30000
rtp_port_end = 40000
```

任务：

- 新增统一 `main.cpp`。
- 支持加载统一配置。
- 固定创建 `GB28181Node` 并装配完整能力模块。
- 统一日志、信号处理、生命周期管理和退出流程。
- 迁移完成后移除旧的两个 main 作为正式入口。

验收标准：

- 单一二进制启动后同时具备上级能力和下级能力。
- 上下级关系在同一节点内由能力模块共同处理，端口、配置、日志和会话归属不混淆。
- 不存在按上级/下级能力裁剪启动的运行模式。

## 里程碑 6：现有链路验证

目标：证明整理后的工程仍能跑通当前已有功能。

验证链路：

```text
注册客户端与媒体发送能力
  -> REGISTER
  -> Keepalive MESSAGE
  -> Catalog Response
  -> INVITE 200 OK
  -> H.264 -> PS Mux -> RTP Send

注册服务端、目录查询、开流和媒体接收能力
  -> REGISTER Auth/Response
  -> Keepalive Handle
  -> Catalog Query
  -> INVITE
  -> RTP Receive -> PS Demux -> H.264 Dump
```

验收标准：

- 能完成上下级注册。
- 能完成心跳保活。
- 能完成目录查询。
- 能发起实时预览 INVITE。
- 能接收 RTP/PS 并输出可验证的 H.264 裸流。
- 使用 `ffplay` 验证裸流：

```bash
ffplay -f h264 output.h264
```
