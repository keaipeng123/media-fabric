# 阶段 0：整理现有项目基础执行计划

更新日期：2026-06-29

## 目标

将当前 `GB28181-Server` 整理成后续流媒体项目式学习的工程底座，并把 `SipSupService` 与 `SipSubService` 逐步整合为一个统一项目：

```text
gb28181-server
  -> sup role：上级平台能力
  -> sub role：下级平台/设备模拟能力
  -> both role：同一进程同时启用上下级能力
```

阶段 0 不追求新增 FFmpeg、SRS、WebRTC 等能力，重点是让现有项目结构、构建、文档和验证链路变清楚。

## 当前整理结果

- 项目学习计划已迁移到 `docs/流媒体开发项目式学习计划.md`。
- 原 `notes` 中长期维护的知识笔记已迁移到 `docs/notes/`。
- 原 `notes/days` 中 day 日志属于过程记录，已从维护文档中移除。

## 执行原则

- 先统一工程入口，再重构业务代码。
- 先保持行为不变，再抽取公共模块。
- 先保留 `SipSupService` 和 `SipSubService` 的原有可执行目标，验证稳定后再收敛到单一二进制。
- 所有阶段性成果都要能被 README、流程图、抓包或命令验证。

## 里程碑 2：统一构建入口

目标：先让工程具备统一构建入口，但暂时仍保留两个旧服务目标。

任务：

- 在根目录新增 `CMakeLists.txt`。
- 修复子工程 CMake：
  - 补齐 `cmake_minimum_required()`。
  - 补齐 `project()`。
  - 明确 C++ 标准。
  - 整理 include 和 link 目录。
- 让统一构建先生成：
  - `SipSupService`
  - `SipSubService`
- 明确 Linux 构建环境，因为当前第三方库以 `x86_64-unknown-linux-gnu` 为主。

验收标准：

```bash
cmake -S . -B build
cmake --build build
```

能在 Linux 环境下完成构建，并生成两个原有服务目标。

## 里程碑 3：抽取公共模块

目标：把上下级重复代码收敛到公共库，为单服务入口做准备。

候选公共模块：

```text
common/
  ConfReader
  ECSocket
  ECThread
  ECEventPoll
  ThreadPool
  TaskTimer
  XmlParser
  SipMessage
  SipTaskBase
```

任务：

- 对比 `SipSupService` 与 `SipSubService` 中同名文件差异。
- 先抽取完全一致或差异很小的模块。
- 建立 `gb28181_common` 静态库。
- 原 `SipSupService` 和 `SipSubService` 改为链接公共库。

验收标准：

- 抽取后两个旧服务行为不变。
- 公共模块不再在两个服务中重复维护。
- 每次抽取后能独立构建验证。

## 里程碑 4：角色化封装

目标：把上级能力和下级能力封装成可被同一进程启动的角色模块。

建议结构：

```text
roles/
  SupServiceRole
  SubServiceRole
```

任务：

- 将 `SipSupService/src/main.cpp` 中的启动流程封装为 `SupServiceRole`。
- 将 `SipSubService/src/main.cpp` 中的启动流程封装为 `SubServiceRole`。
- 每个角色维护自己的配置、SIP 监听端口、PJSIP 上下文、线程和定时器。
- 避免上下级共用同一个全局 SIP 状态。

验收标准：

- 仍能分别以 sup/sub 方式启动。
- 上下级端口隔离清晰，例如上级 `5061`，下级 `7101`。
- 角色初始化、启动、停止边界清楚。

## 里程碑 5：单一二进制入口

目标：新增统一入口 `gb28181-server`，编译时生成一个服务，运行时决定启用角色。

建议启动方式：

```bash
./gb28181-server --role sup
./gb28181-server --role sub
./gb28181-server --role both
```

或配置方式：

```ini
[role]
enable_sup = true
enable_sub = true
```

任务：

- 新增统一 `main.cpp`。
- 支持加载统一配置。
- 根据 role 启动 `SupServiceRole`、`SubServiceRole` 或两者。
- 保留原服务目标一段时间作为对照验证。

验收标准：

- 单一二进制可以启动上级能力。
- 单一二进制可以启动下级能力。
- 单一二进制可以同时启动上下级能力，且端口、配置、日志互不混淆。

## 里程碑 6：现有链路验证

目标：证明整理后的工程仍能跑通当前已有功能。

验证链路：

```text
Sub role
  -> REGISTER
  -> Keepalive MESSAGE
  -> Catalog Response
  -> INVITE 200 OK
  -> H.264 -> PS Mux -> RTP Send

Sup role
  -> REGISTER Auth/Response
  -> Keepalive Handle
  -> Catalog Query
  -> INVITE
  -> RTP Receive -> PS Demux -> H.264 Dump
```

验收标准：

- 能完成上下级注册。
- 能完成心跳保活。
- 能完成目录查询。
- 能发起实时预览 INVITE。
- 能接收 RTP/PS 并输出可验证的 H.264 裸流。
- 使用 `ffplay` 验证裸流：

```bash
ffplay -f h264 output.h264
```
