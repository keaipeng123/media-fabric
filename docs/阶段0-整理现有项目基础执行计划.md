# 阶段 0：整理现有项目基础执行计划

更新日期：2026-07-01

## 目标

将当前 `GB28181-Server` 整理成后续流媒体项目式学习的工程底座，并把 `SipSupService` 与 `SipSubService` 逐步整合为一个统一项目、一个进程、一个服务入口：

```text
gb28181-server
  -> GB28181Node
      -> SipStack
      -> PeerRegistry
      -> RegisterClientCapability
      -> RegisterServerCapability
      -> KeepaliveCapability
      -> CatalogCapability
      -> InviteCapability
      -> MediaReceiveCapability
      -> MediaSendCapability
```

最终工程标准不是运行时选择启用哪些能力，而是 `gb28181-server` 进程天然具备 GB28181 节点应有的完整能力。上级/下级只表示某条级联关系中的相对身份：对上级平台，本节点表现为注册客户端；对下级平台或设备，本节点表现为注册服务端。代码组织应按能力拆分，而不是按上下级身份拆分。

阶段 0 不追求新增 FFmpeg、SRS、WebRTC 等能力，重点是让现有项目结构、构建、文档和验证链路变清楚。

## 执行原则

- 先统一工程入口，再重构业务代码。
- 先保持行为不变，再抽取公共模块。
- `SipSupService` 和 `SipSubService` 只作为历史代码来源和迁移参照，不作为最终交付形态。
- 阶段 0 的工程主线始终指向单一进程：一个配置、一个日志体系、一个 `GB28181Node`、一组能力模块、一个服务入口。
- 能力模块需要能被同一个节点同时装配，例如注册客户端和注册服务端可以同时存在，媒体接收和媒体发送也可以同时存在。
- 所有阶段性成果都要能被 README、流程图、抓包或命令验证。

## 里程碑 5：单一二进制入口

```bash
./gb28181-server -c conf/gb28181-server.conf
```

```ini
[node]

```

## 里程碑 6：现有链路验证

目标：证明整理后的工程仍能跑通当前已有功能。

验证链路：

```text
注册客户端与媒体发送能力
  -> REGISTER
  -> Keepalive MESSAGE
  -> Catalog Response
  -> INVITE 200 OK
  -> H.264 -> PS Mux -> RTP Send

注册服务端、目录查询、开流和媒体接收能力
  -> REGISTER Auth/Response
  -> Keepalive Handle
  -> Catalog Query
  -> INVITE
  -> RTP Receive -> PS Demux -> H.264 Dump
```

验收标准：

- 能完成上下级注册。
- 能完成心跳保活。
- 能完成目录查询。
- 能发起实时预览 INVITE。
- 能接收 RTP/PS 并输出可验证的 H.264 裸流。
- 使用 `ffplay` 验证裸流：

```bash
ffplay -f h264 output.h264
```
